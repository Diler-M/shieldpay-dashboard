Application Summary

ShieldPay is a payment processing and merchant management platform built with Express backend and React frontend. It handles user authentication, merchant accounts, customer management, card processing, payment transactions, and provides admin/analytics capabilities with JWT-based security.

Technology Stack

Express.jsReactSQLiteJWTbcryptReact RouterVitestVite

Data Flows

  • Data enters via REST API endpoints (registration, login, payment processing, customer/card management)
  • Authentication via JWT tokens stored in httpOnly cookies; password hashing with bcrypt
  • Data processed through Express route handlers with rate limiting and permission checks
  • Data persisted to SQLite database (user accounts, merchants, customers, cards, transactions, API keys)
  • Data exits via API responses, exports, webhooks, and admin reporting endpoints

Attack Surface

  • Authentication endpoints: /register, /login, /logout, /reset-password (21 input validation issues detected)
  • Admin impersonation endpoint: POST /impersonate/:id (no additional authorization checks)
  • Payment processing: POST /process (handles sensitive transaction data)
  • API key management: GET/POST /api-keys (credential exposure risk)
  • Data export: GET /export (potential information disclosure)
  • Webhook handling: GET /webhooks (URL validation concerns)
  • User enumeration via timing attacks on login/registration
  • Rate limiting bypass vectors on registration and password reset

Impact: Platform abuse through mass fake account creation. Successful brute force of password reset tokens leading to account takeover. Resource exhaustion causing service degradation for legitimate users. Increased fraud risk from disposable attacker accounts. Moderate to high impact: account compromise and platform abuse.
Attack Vector: 1) Identify rate limiting implementation weaknesses (IP-based only, no distributed rate limiting). 2) Use proxy rotation or VPN services to bypass IP-based limits. 3) Manipulate X-Forwarded-For headers if trusted by application. 4) Automate registration of hundreds of fake merchant accounts. 5) Brute force password reset tokens within rate limit windows. 6) Use compromised accounts for fraudulent transactions.
JWT Token Manipulation and Privilege Escalation Mitigated

Attacker attempts to manipulate JWT tokens to escalate privileges from merchant to admin role. By exploiting weak JWT validation, missing signature verification, or algorithm confusion attacks (e.g., switching from RS256 to HS256), attacker modifies role claims in the token to gain unauthorized admin access.

Impact: Unauthorized access to admin functionality including user impersonation, data export, and platform-wide analytics. Ability to view and modify all merchant accounts. Access to sensitive configuration and API keys. High impact: complete authorization bypass and potential data breach.
Attack Vector: 1) Capture valid JWT token from authenticated merchant session. 2) Decode JWT and analyze structure and claims. 3) Attempt algorithm confusion attack (none algorithm, HS256 with public key). 4) Modify role claim from 'merchant' to 'admin'. 5) Re-encode token and submit with modified claims. 6) If successful, access admin endpoints like /impersonate, /export, and admin analytics.
SQL Injection via Input Validation Gaps 1 vulnerabilities found

Attacker exploits the 21 detected input validation issues across authentication and data management endpoints to inject SQL commands. SQLite database is particularly vulnerable to injection attacks that can read arbitrary data, modify records, or extract the entire database through UNION-based or blind injection techniques.

Impact: Unauthorized data access including payment information, customer PII, and credentials. Ability to modify transaction records or user permissions. Potential for complete database extraction. High impact: data breach, financial fraud, and compliance violations.
Attack Vector: 1) Identify endpoints with input validation issues (registration, login, customer/card management). 2) Test for SQL injection using SQLite-specific payloads. 3) Exploit injection points to extract data using UNION SELECT or blind injection. 4) Target sensitive tables (users, transactions, cards, api_keys). 5) Exfiltrate data through error messages or timing-based techniques. 6) Modify transaction records or elevate account privileges through UPDATE/INSERT injection.
Session Fixation and Cookie Theft 4 vulnerabilities found

Attacker exploits JWT session management with cookie fallback to perform session fixation or steal session tokens. By intercepting httpOnly cookies through XSS vulnerabilities, man-in-the-middle attacks, or subdomain attacks, attacker hijacks authenticated sessions to impersonate legitimate users.

Impact: Account takeover of merchant and admin accounts. Unauthorized access to payment processing and customer data. Ability to perform transactions as compromised user. Moderate to high impact depending on compromised account privileges.
Attack Vector: 1) Identify XSS vulnerabilities in React frontend or API responses. 2) Inject malicious script to exfiltrate JWT tokens from cookies or localStorage. 3) Alternatively, perform session fixation by forcing victim to use attacker-controlled session token. 4) Use stolen/fixed session to authenticate as victim. 5) Access merchant dashboard, process payments, or export customer data. 6) For admin sessions, gain platform-wide access.

Vulnerability Details

CRITICAL Admin Impersonation Endpoint - Unauthorized Privilege Escalation
security-report-2026-04-13_Phase-4.html:297

The /impersonate/:id endpoint allows any authenticated user to impersonate admin accounts without proper authorization checks. The endpoint returns admin JWT tokens in JSON responses, enabling complete platform takeover.

Recommendation: Add role-based authorization check before impersonation: verify current user has 'admin' role. Implement audit logging for all impersonation attempts. Consider removing endpoint entirely or restricting to super-admin only with MFA requirement.
CRITICAL Deterministic Payment Status Based on Amount - Transaction Manipulation
shieldpay/backend/routes/payments.js:72

Payment approval/decline is determined solely by amount comparison (>1000 = DECLINED, <=1000 = APPROVED) without any actual payment processing, card validation, or fraud checks. This is a mock payment processor that allows attackers to reliably predict and manipulate transaction outcomes.

Recommendation: Integrate real payment gateway (Stripe, PayPal, etc.) with proper authorization flows. Remove deterministic status logic. Implement fraud detection, card validation, and proper payment processing workflows.
CRITICAL Insufficient Authorization Check on Impersonation Endpoint
shieldpay/backend/routes/auth.js:169

The /impersonate/:id endpoint only verifies admin role but lacks additional authorization controls. Any user who achieves admin role (via JWT manipulation, privilege escalation, or compromised credentials) can impersonate any user through sequential ID enumeration.

Recommendation: Implement additional authorization: require MFA re-authentication for impersonation, log all impersonation attempts with alerting, restrict impersonation to specific user roles, implement time-limited impersonation tokens, and add approval workflow for sensitive impersonations.
HIGH Full Card PAN Processed Client-Side - Data Exposure Risk
shieldpay/frontend/src/pages/NewPaymentPage.jsx:46

Full card PAN (c.pan) is passed to client-side JavaScript function maskPanForDisplay(). While masked for display, the full PAN is accessible in browser memory, developer tools, and network responses, violating PCI-DSS requirements.

Recommendation: Backend should only return masked PAN (e.g., last 4 digits) to frontend. Store card_id for payment processing, never send full PAN to client. Use tokenization for card references.
HIGH Full PAN Exposure in Transaction API Response
shieldpay/backend/routes/transactions.js:43

The GET /:id endpoint returns full card PAN (Primary Account Number) in transaction responses via 'cards.pan' field. This violates PCI-DSS requirement 3.3 which prohibits displaying full PAN except for specific business needs.

Recommendation: Remove 'cards.pan' from SELECT query. Only return 'cards.pan_last4' for display purposes. If full PAN is needed for specific operations, create separate privileged endpoint with additional authorization and audit logging.
HIGH SQL Injection in Registration Endpoint - merchantName Parameter
shieldpay/backend/routes/auth.js:137

The /register endpoint accepts merchantName parameter and passes it directly to SQL INSERT without proper parameterization. While the code uses db.prepare(), the merchantName value is concatenated into the query string.

Recommendation: Use parameterized queries with placeholders: db.prepare('INSERT INTO merchants (name) VALUES (?)').run(merchantName). Add input validation for merchantName (length limits, character whitelist). Implement prepared statements consistently across all database operations.
HIGH In-Memory Rate Limiting Bypass via Server Restart or Distributed Deployment
shieldpay/backend/routes/payments.js:52

Payment rate limiting uses in-memory Map storage without persistence or distributed coordination. Rate limit state is lost on server restart, process crash, or in multi-instance deployments, allowing attackers to bypass limits entirely.

Recommendation: Implement distributed rate limiting using Redis or similar shared state store. Persist rate limit counters with TTL. Use rate limiting middleware that works across multiple instances (e.g., express-rate-limit with Redis store).
HIGH No Rate Limiting on Registration Endpoint
shieldpay/backend/routes/auth.js:137

The /register endpoint lacks rate limiting controls. Combined with X-Forwarded-For spoofing (finding #1), attackers can create unlimited merchant accounts for resource exhaustion, account enumeration preparation, and fraudulent transaction staging.

Recommendation: Implement rate limiting on registration (e.g., 5 registrations per IP per hour), add CAPTCHA verification, require email verification before account activation, implement phone verification for merchant accounts, and monitor for bulk registration patterns.
HIGH X-Forwarded-For Header Spoofing Enables Rate Limit Bypass
shieldpay/backend/routes/auth.js:16

The clientIp() function unconditionally trusts the X-Forwarded-For header without validation. An attacker can spoof this header to bypass IP-based rate limiting on authentication endpoints, enabling brute force attacks and account enumeration.

Recommendation: Only trust X-Forwarded-For when behind a verified proxy. Validate the header format and use the rightmost trusted IP. Implement distributed rate limiting using Redis/Memcached keyed by user identifier (email) in addition to IP.

MEDIUM Timing Attack on Login Endpoint Enables User Enumeration
shieldpay/backend/routes/auth.js:104

The /login endpoint performs bcrypt.compareSync() only when user exists, creating measurable timing differences. When user doesn't exist (line 116-121), response is immediate. When user exists but password wrong (line 122-127), bcrypt comparison takes ~100ms. This timing difference leaks user existence.

Recommendation: Always perform bcrypt comparison even for non-existent users: generate a dummy hash and compare against provided password. This ensures constant-time response regardless of user existence. Alternatively, use a constant-time comparison delay.
LOW Missing Validation on API Key Name Enables Stored XSS

Recommendation: 1) Validate API key name (alphanumeric, spaces, hyphens only), 2) Limit length to reasonable value (e.g., 50 chars), 3) Escape output in frontend when rendering API key names.
