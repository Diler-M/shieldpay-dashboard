Application Summary

ShieldPay is a payment processing and merchant management platform built with Express backend and React frontend. It handles user authentication, merchant accounts, customer management, transaction processing, and payment card management with JWT-based authorization.

Technology Stack

Express.jsReactSQLiteJWTbcryptReact RouterVitestbetter-sqlite3

Data Flows

  • Data enters via REST API endpoints (login, registration, payment processing, customer/transaction CRUD)
  • Authentication via JWT tokens (stored in localStorage on client, validated on backend)
  • Data processed through route handlers with minimal input validation
  • Data stored in SQLite database (user credentials, customer records, transactions, payment cards)
  • Data exits via API responses, exports, and webhooks

Attack Surface

  • Authentication endpoints: /login, /register, /reset-password, /impersonate/:id (36 total API endpoints)
  • User input processing: email, password, customer data, transaction queries without consistent validation
  • Authorization checks: 9 permission checks across 8 files, but multiple IDOR vulnerabilities detected
  • Session management: JWT tokens in localStorage (vulnerable to XSS)
  • Admin operations: /users, /merchants endpoints lack role-based access control
  • Search functionality: SQL injection vulnerability in customer search

Threat Model

Worst Case Scenarios

Complete Financial Data Breach 9 vulnerabilities found

Attacker gains full access to all payment card data, transaction records, and customer financial information across all merchants. This includes card numbers, transaction histories, merchant banking details, and customer PII.

Impact: Massive financial liability, PCI DSS violations, regulatory fines, complete loss of customer trust, potential bankruptcy from lawsuits and compliance penalties. All merchants lose their payment processing capabilities.
Attack Vector: SQL injection in customer search combined with privilege escalation through IDOR vulnerabilities to access admin functions, then dump entire SQLite database containing unencrypted payment data.
Mass Account Takeover Campaign 4 vulnerabilities found

Attacker compromises admin accounts and uses impersonation endpoint to systematically take over all merchant and customer accounts, conducting fraudulent transactions and stealing funds.

Impact: Complete loss of platform integrity, massive financial theft, regulatory shutdown, criminal liability for facilitating fraud. Platform becomes unusable and untrustworthy.
Attack Vector: XSS attack steals admin JWT from localStorage, then uses /impersonate/:id endpoint without proper authorization checks to assume any user identity and conduct fraudulent operations.
Payment Processing Infrastructure Compromise 2 vulnerabilities found

Attacker gains control of payment processing logic and webhook systems to redirect all transaction funds to attacker-controlled accounts while maintaining appearance of normal operations.

Impact: Silent theft of all processed payments, delayed detection allows massive financial losses, complete breakdown of merchant trust, potential criminal charges for operators.
Attack Vector: Exploit weak JWT validation and lack of role-based access control on admin endpoints to modify payment routing configurations and webhook destinations.

Likely Attack Scenarios

SQL Injection Data Extraction Mitigated

Attacker exploits the identified SQL injection vulnerability in customer search to extract sensitive customer data, payment information, and user credentials from the SQLite database.

Impact: Data breach affecting customer PII and payment data, potential PCI compliance violations, customer notification requirements, reputation damage.
Attack Vector: Craft malicious SQL payloads in customer search parameters to bypass input validation and extract data through UNION-based or blind SQL injection techniques.
JWT Token Theft via XSS Mitigated

Attacker injects malicious JavaScript to steal JWT tokens from localStorage, gaining unauthorized access to user accounts and sensitive operations.

Impact: Account compromise, unauthorized transactions, data theft, potential financial losses for affected users.
Attack Vector: Exploit lack of input validation on user-controlled fields to inject XSS payloads that steal JWT tokens from localStorage and transmit them to attacker-controlled servers.
Insecure Direct Object Reference Exploitation 4 vulnerabilities found

Attacker manipulates object references in API calls to access other users' data, transactions, and account information without proper authorization checks.

Impact: Unauthorized access to sensitive customer and transaction data, privacy violations, potential financial fraud.
Attack Vector: Enumerate user IDs and object references in API endpoints, bypassing insufficient authorization checks to access data belonging to other users or merchants.
Credential Stuffing Attack Mitigated

Attacker uses automated tools to test compromised credentials from other breaches against the login endpoint, exploiting lack of rate limiting and account lockout mechanisms.

Impact: Multiple account compromises, unauthorized access to merchant dashboards, potential financial theft and data exposure.
Attack Vector: Use botnets to rapidly test credential combinations against /login endpoint, exploiting absence of rate limiting, CAPTCHA, or account lockout protections.
Admin Panel Unauthorized Access Mitigated

Attacker exploits weak role-based access controls to gain unauthorized access to admin functions like user management and merchant operations.

Impact: Privilege escalation, ability to modify user accounts, access to sensitive administrative data, potential for further system compromise.
Attack Vector: Manipulate JWT tokens or exploit missing role validation on admin endpoints (/users, /merchants) to access administrative functions without proper authorization.

Vulnerability Details

CRITICAL SQL Injection in Customer Search
shieldpay/backend/routes/customers.js:12

User-supplied search query is directly concatenated into SQL string without parameterization. The req.query.q parameter is embedded directly into the SQL query using string concatenation.

Recommendation: Use parameterized queries: db.prepare('SELECT * FROM customers WHERE merchant_id = ? AND name LIKE ?').all(req.user.merchant_id, `%${query}%`)
CRITICAL CORS Misconfiguration - Wildcard Origin with Credentials
shieldpay/backend/server.js:25

CORS is configured with app.use(cors()) without any origin restrictions. This allows any external website to make authenticated requests to the API if the user visits a malicious site while logged in.

Recommendation: Configure CORS with specific allowed origins, not wildcard
CRITICAL Missing role-based access control on /admin/merchants endpoint
shieldpay/backend/routes/admin.js:14

The GET /merchants endpoint returns all merchants in the system without verifying that the authenticated user has admin role. Any authenticated user can access this sensitive endpoint.

Recommendation: Add role check to verify req.user.role === 'admin' before returning data
CRITICAL Missing role-based access control on /admin/users endpoint
shieldpay/backend/routes/admin.js:8

The GET /users endpoint returns all users in the system without verifying that the authenticated user has admin role. Any authenticated user can access this sensitive endpoint.

Recommendation: Add role check to verify req.user.role === 'admin' before returning data
CRITICAL Missing Authorization Check on Impersonate Endpoint
shieldpay/backend/routes/auth.js:52

The /impersonate/:id endpoint only checks if the user is authenticated but does not verify if the authenticated user has permission to impersonate other users. Any authenticated user can impersonate any other user including admins.

Recommendation: Add role-based authorization check to verify only admins can impersonate
CRITICAL Full Payment Card Data Stored in Plaintext
shieldpay/backend/db.js:98

Complete payment card data (PAN, CVV, expiry) is stored in plaintext in the SQLite database. This violates PCI DSS requirements and exposes sensitive financial data.

Recommendation: Encrypt or tokenize payment card data, never store CVV
CRITICAL Sensitive Payment Card Data Exposure in Frontend
shieldpay/frontend/src/pages/TransactionDetailPage.jsx:20

The transaction detail page displays unencrypted PAN and CVV directly in the UI. These are highly sensitive payment card details that should never be displayed in plaintext.

Recommendation: Never display full PAN or CVV - mask PAN and remove CVV display entirely
CRITICAL IDOR - Missing merchant_id ownership check on PUT card update
shieldpay/backend/routes/cards.js:31

The PUT /:id endpoint updates payment card details without merchant ownership verification. Code comment explicitly notes this vulnerability.

Recommendation: Add merchant_id filter to UPDATE query and verify ownership before update
CRITICAL IDOR - Missing merchant_id ownership check on GET card details
shieldpay/backend/routes/cards.js:14

The GET /:id endpoint retrieves payment card details without merchant ownership verification, exposing PAN, CVV, and expiry data.

Recommendation: Add merchant_id filter: WHERE id = ? AND merchant_id = ?
HIGH JWT Token Stored in Vulnerable localStorage
shieldpay/frontend/src/pages/LoginPage.jsx:16

JWT authentication tokens are stored in localStorage, which is vulnerable to XSS attacks. Any XSS vulnerability in the application allows attackers to steal the token and impersonate the user.

Recommendation: Use httpOnly cookies for token storage instead of localStorage
HIGH Sensitive Payment Card Data Exposure in DOM
shieldpay/frontend/src/pages/CardsPage.jsx:35

Payment card PAN and CVV values are rendered directly in the table without any masking. This exposes sensitive payment card data in the DOM where it can be accessed via browser DevTools or XSS attacks.

Recommendation: Mask PAN display and never show CVV in UI
HIGH Stack Trace and Request Body Exposure in Error Responses
shieldpay/backend/server.js:76

The error handler returns full stack traces and request bodies to clients in all environments. This leaks sensitive information including database structure, file paths, and potentially credentials/card data from failed requests.

Recommendation: Return generic error messages in production and log detailed errors server-side only
HIGH Sensitive Data Exposure in Request Logging
shieldpay/backend/server.js:37

The request logging middleware logs all request bodies without filtering. This captures passwords during login/registration, payment card data, and other sensitive information to console logs.

Recommendation: Filter sensitive fields from request logging or disable body logging in production
HIGH Weak Default JWT Secret
shieldpay/backend/middleware/auth.js:4

JWT secret falls back to hardcoded 'shieldpay-dev-secret' if JWT_SECRET environment variable is not set. This allows attackers to forge valid JWT tokens if the application is deployed without proper environment configuration.

Recommendation: Remove fallback secret and require JWT_SECRET environment variable to be set
