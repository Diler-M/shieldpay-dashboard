Application Summary

ShieldPay is a payment processing platform built with Express backend and React frontend. It manages user authentication, merchant accounts, payment transactions, card management, and customer data with JWT-based authorization and SQLite persistence.

Technology Stack

Express.jsReactSQLiteJWTbcryptReact RouterVitestAxios

Data Flows

  • Data enters via REST API endpoints (login, registration, payment processing, card management)
  • Authentication via JWT tokens and API keys; user credentials hashed with bcrypt
  • Data stored in SQLite database with user accounts, transactions, cards, and customer records
  • Data exits via API responses, exports, and webhooks; sensitive tokens exposed in JSON responses

Attack Surface

  • Authentication endpoints (/login, /register, /reset-password) - protected with rate limiting and token-response hardening
  • Impersonation endpoint (/impersonate/:id) - admin-gated, cookie-only auth token delivery, and audit logged
  • Payment processing endpoint (/process) - handles financial transactions with ownership validation and throttling
  • API key management endpoints - masked key exposure and cryptographic key generation
  • Admin endpoints (/users, /merchants) - role-gated
  • Card management endpoints - handles payment instrument data with masked storage and ownership checks
  • Input validation and abuse controls are applied across critical routes

Threat Model

Worst Case Scenarios

Complete Financial Data Breach via SQL Injection 1 vulnerabilities found

Attacker exploits missing input validation across multiple routes to perform SQL injection attacks against the SQLite database, extracting all user credentials, payment card data, transaction history, merchant accounts, and customer records. The breach exposes sensitive financial information for all platform users.

Impact: Complete loss of customer trust, massive financial liability from exposed payment card data, regulatory fines under PCI-DSS and data protection laws (GDPR/CCPA), class-action lawsuits, potential business closure. Technical impact includes full database compromise with all historical transaction data exposed.
Attack Vector: Attacker identifies endpoints with missing input validation (payment processing, card management, customer endpoints) and crafts malicious SQL payloads in request parameters. SQLite's lack of parameterized query enforcement makes injection trivial if developers used string concatenation. Attacker extracts data incrementally or dumps entire database.
Admin Account Takeover via Impersonation Endpoint Abuse Mitigated

The impersonation path now enforces admin-role authorization, validates target IDs, emits audit logs, and does not return auth tokens in JSON responses. Authentication is delivered via httpOnly cookie only, significantly reducing takeover risk from response-token leakage.

Impact: Complete platform compromise with attacker having persistent admin access. Ability to manipulate financial transactions, steal funds, access all merchant and customer data, modify audit logs to hide tracks. Reputational destruction and potential criminal liability for the business. Loss of all merchant accounts as trust evaporates.
Attack Vector: Residual risk is primarily insider misuse by legitimate admins rather than unauthenticated privilege escalation. Endpoint use is constrained by role checks, input validation, and audit events, and no JWT is exposed in the response body.
Mass Payment Fraud via Authentication Bypass 2 vulnerabilities found

Attacker discovers authentication bypass vulnerability in JWT validation or exploits weak bcrypt configuration to crack passwords at scale. With compromised merchant accounts, attacker initiates fraudulent payment transactions, redirects funds to attacker-controlled accounts, and manipulates transaction records to hide the fraud.

Impact: Direct financial theft potentially affecting thousands of transactions. Merchant accounts drained of funds. Platform liable for fraudulent transactions. Immediate loss of payment processor partnerships and merchant trust. Emergency shutdown required, causing business interruption. Potential criminal investigation and regulatory sanctions.
Attack Vector: Attacker exploits weak JWT secret (if hardcoded or predictable), JWT algorithm confusion attacks (switching RS256 to HS256), or missing signature verification. Alternatively, uses credential stuffing on /login endpoint (no rate limiting mentioned) or exploits password reset flow. Once authenticated as merchant, uses /process endpoint to create fraudulent transactions with manipulated amounts or recipient accounts.
Supply Chain Attack via Dependency Compromise Mitigated

Attacker compromises a dependency in the Express.js, React, or npm package ecosystem used by ShieldPay. Malicious code is injected that exfiltrates JWT tokens, API keys, database credentials, and payment card data to attacker-controlled servers. The compromise affects all platform users and goes undetected for extended periods.

Impact: Silent data exfiltration over weeks or months before detection. All authentication tokens, API keys, and sensitive data compromised. Attacker maintains persistent backdoor access. Massive breach disclosure affecting all users. Complete loss of platform credibility. Potential regulatory action for inadequate security controls. Expensive incident response and forensic investigation.
Attack Vector: Attacker compromises a popular npm package used by ShieldPay (Axios, bcrypt, or middleware packages) through account takeover of package maintainer or typosquatting. Malicious version includes code that intercepts JWT tokens from httpOnly cookies, captures API keys from headers, and exfiltrates SQLite database contents. Code is obfuscated to avoid detection during code reviews.

Likely Attack Scenarios

Brute Force Attack on Authentication Endpoints 1 vulnerabilities found

Attacker launches automated brute force or credential stuffing attacks against /login and /register endpoints. Without rate limiting or account lockout mechanisms, attacker systematically tests compromised credentials from data breaches or common password lists to gain unauthorized access to merchant and admin accounts.

Impact: Unauthorized access to merchant accounts leading to fraudulent transactions, data theft, and account takeover. Customer data exposure. Increased infrastructure costs from attack traffic. Legitimate users locked out if attacker changes credentials. Damage to platform reputation.
Attack Vector: Attacker uses automated tools (Hydra, Burp Intruder) to send thousands of login requests with credential lists. Lack of rate limiting, CAPTCHA, or account lockout allows unlimited attempts. Successful compromises grant access to merchant dashboards and payment processing capabilities. Attacker may also enumerate valid usernames through timing attacks or error message differences.
JWT Token Theft via XSS or Insecure Storage 4 vulnerabilities found

Attacker exploits missing input validation to inject malicious JavaScript that steals JWT tokens from localStorage or intercepts them from API responses. The stolen tokens are used to impersonate users and access protected resources without authentication.

Impact: Session hijacking allowing attacker to impersonate legitimate users. Access to merchant accounts, transaction history, and customer data. Ability to perform actions as the compromised user including initiating payments. Users unaware their accounts are compromised until fraudulent activity detected.
Attack Vector: Attacker injects XSS payload through unvalidated input fields (customer names, transaction descriptions, merchant profiles). Malicious script executes in victim's browser, stealing JWT from localStorage or intercepting API responses containing tokens. If tokens stored in localStorage (common React pattern), simple document.cookie or localStorage access exfiltrates credentials to attacker server.
API Key Leakage and Abuse 4 vulnerabilities found

Long-lived API keys managed through API key management endpoints are exposed through insecure storage, logging, or client-side code. Attacker discovers these keys and uses them to bypass authentication, access merchant data, and perform unauthorized operations.

Impact: Persistent unauthorized access to platform APIs. Ability to query sensitive data, initiate transactions, and access merchant accounts without user credentials. Difficult to detect as API key usage may appear legitimate. Potential for automated abuse and data scraping. Revocation requires coordinating with all legitimate API key holders.
Attack Vector: Attacker finds API keys hardcoded in React frontend code, exposed in GitHub repositories, leaked in error logs, or transmitted over unencrypted connections. Uses discovered keys to authenticate API requests. Alternatively, exploits API key management endpoints to generate new keys or steal existing ones if authorization checks are weak.
Authorization Bypass via IDOR on Merchant Endpoints 2 vulnerabilities found

Attacker manipulates resource identifiers in API requests to access other merchants' data. Weak authorization checks on admin and merchant endpoints allow horizontal privilege escalation where one merchant can view or modify another merchant's transactions, customers, and payment data.

Impact: Unauthorized access to competitor merchant data including transaction volumes, customer lists, and payment patterns. Data privacy violations exposing customer information across merchant boundaries. Ability to modify other merchants' data causing business disruption. Regulatory violations under data protection laws.
Attack Vector: Attacker authenticates as legitimate merchant, then modifies merchant_id or user_id parameters in API requests to admin endpoints (/users, /merchants) or merchant data endpoints. If authorization only checks authentication but not resource ownership, attacker accesses other merchants' data. Systematic enumeration of IDs reveals all platform data.
Password Reset Flow Exploitation 3 vulnerabilities found

Attacker exploits vulnerabilities in the /reset-password endpoint such as missing token expiration, predictable reset tokens, or lack of rate limiting to take over user accounts. Attacker can reset passwords for arbitrary users including admin accounts.

Impact: Account takeover of merchant and admin accounts. Loss of account access for legitimate users. Unauthorized access to financial data and transaction capabilities. Potential for fraudulent transactions and data theft. Customer support burden handling account recovery requests.
Attack Vector: Attacker requests password reset for target account. If reset tokens are predictable (sequential, timestamp-based) or never expire, attacker can guess or brute force valid tokens. Alternatively, exploits lack of rate limiting to request multiple resets and test tokens. If email verification is weak or bypassable, attacker completes reset and gains account access.

CRITICAL API Keys Exposed in Frontend Code
shieldpay/frontend/src/pages/SettingsPage.jsx:37

API key values are rendered directly in the DOM as plaintext within <code> tags. These long-lived credentials are visible in browser DevTools, page source, and can be stolen through XSS attacks or browser extensions. The keys are transmitted in API responses and stored in React component state, creating multiple exposure vectors.

Recommendation: 1) Never display full API key values in the frontend - show only last 4 characters (e.g., 'sk_live_****1234'), 2) Implement 'reveal' functionality that requires re-authentication and shows keys only temporarily, 3) Add audit logging when API keys are viewed, 4) Implement API key rotation and expiration policies, 5) Use short-lived tokens instead of long-lived API keys where possible, 6) Add CSP headers to prevent XSS-based key theft.

HIGH JWT Token Exposure Risk via Credentials Configuration
shieldpay/frontend/src/api.js:5

The axios instance is configured with 'withCredentials: true', which sends cookies with all requests. Combined with the application context stating 'JWT tokens exposed in JSON responses' and 'httpOnly cookie fallback', this creates a token exposure risk. HIGH XSS via Unsanitized Customer Name in Transaction List
shieldpay/frontend/src/pages/TransactionsPage.jsx:20

The customer_name field from API response is rendered directly in JSX without sanitization. HIGH XSS via Unescaped Error Message Rendering
shieldpay/frontend/src/pages/LoginPage.jsx:32

Error messages from API responses are rendered directly into the DOM without sanitization. The error variable comes from err.response?.data?.message, which is server-controlled but may contain user-influenced content if the backend echoes user input in error messages. React's JSX provides default escaping for text content, but if the backend returns HTML or the error is rendered in a context where JSX escaping doesn't apply, XSS is possible.

Recommendation: 1) Sanitize all error messages on the backend before sending to frontend - never echo user input directly, 2) Use DOMPurify or similar library to sanitize error messages before rendering, 3) Implement Content Security Policy (CSP) headers to prevent inline script execution, 4) Use generic error messages that don't include user input (e.g., 'Invalid credentials' instead of 'Invalid email: [user_input]'), 5) Consider using dangerouslySetInnerHTML with sanitization if HTML formatting is needed, but prefer plain text errors.

HIGH Weak Card Token Generation Using Deterministic HMAC
shieldpay/backend/utils/cardSecurity.js:23

Card tokenization uses HMAC-SHA256 with a shared secret to generate panToken from PAN. This is deterministic - same PAN always produces same token. If the secret is compromised, attackers can forge tokens for arbitrary PANs or reverse-engineer PANs from tokens.

Recommendation: 1) Use format-preserving encryption (FPE) or proper tokenization service (Stripe, Adyen), 2) If building custom solution, use non-deterministic tokenization with random IVs, 3) Store token-PAN mapping in separate, highly secured database, 4) Implement token expiry and rotation, 5) Never use HMAC for tokenization - it's designed for authentication, not encryption
